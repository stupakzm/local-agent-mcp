# Plan 02-04: Fix Process Group Kill on Timeout (SAFE-04 Gap Closure)

## Objective

Replace `execFile`'s built-in `timeout` option with a manual `setTimeout` that calls `process.kill(-child.pid, 'SIGTERM')` on Unix, ensuring grandchild processes are killed when a bash command times out.

## Context

VERIFICATION.md identified SAFE-04 as PARTIAL: `bashExec` in `src/tools.ts` sets `detached: true` on Unix to create a process group, but never calls `process.kill(-child.pid, 'SIGTERM')` when timeout fires. Node's `execFile` with `timeout` only kills the direct child, not grandchildren. REQUIREMENTS.md SAFE-04 explicitly requires grandchildren don't survive.

The fix: remove the `timeout` option from `execFile`, add a manual `setTimeout` that fires `process.kill(-child.pid, 'SIGTERM')` on Unix, and track a `timedOut` boolean so the callback resolves with the timeout message. Clear the timer on natural exit.

```typescript
const timer = setTimeout(() => {
  timedOut = true;
  try {
    process.kill(-child.pid!, "SIGTERM");
  } catch {
    // Process may have already exited
  }
}, timeoutMs);

child.on("close", () => {
  clearTimeout(timer);
});
```

4. Update the callback's killed check (line 185): change `if (error && "killed" in error && error.killed)` to also check the `timedOut` flag. The new condition should be:

```typescript
if (timedOut) {
  const seconds = Math.round(timeoutMs / 1000);
  resolve({
    success: false,
    output: `command timed out after ${seconds}s`,
  });
  return;
}
```

Keep the existing `error.killed` check as a fallback for Windows where the built-in timeout is still used.

5. For Windows: keep the `timeout: timeoutMs` in the options but only when `process.platform === "win32"`. Use a conditional spread:

```typescript
{
  cwd: workingDir,
  env,
  maxBuffer: MAX_OUTPUT_BYTES + 1024,
  ...(process.platform !== "win32"
    ? { detached: true }
    : { timeout: timeoutMs }),
}
```

This way Windows still gets the built-in timeout (killing the direct child), and Unix gets the manual timeout with process group kill.

6. Keep the `child.on("error", () => {})` handler to prevent unhandled error events from the kill.

Add this test after the existing "appends warning in full mode" test (line 112):

```typescript
it.skipIf(process.platform === "win32")("kills grandchild processes on timeout", async () => {
  // Spawn a bash command that forks a grandchild sleep process
  // The outer bash prints the grandchild PID then waits
  const result = await executeTool(
    "bash",
    { command: 'bash -c "sleep 60 & echo \\$!; wait"' },
    tempDir,
    "full",
    allowedCommands,
    500, // 500ms timeout — grandchild won't finish in time
  );

  expect(result.success).toBe(false);
  expect(result.output).toContain("timed out");

  // Extract the grandchild PID from stdout (printed before timeout killed the group)
  const pidMatch = result.output.match(/^(\d+)$/m);
  if (pidMatch) {
    const grandchildPid = parseInt(pidMatch[1], 10);
    // Give OS a moment to clean up
    await new Promise((r) => setTimeout(r, 200));
    // Verify grandchild is dead
    let alive = false;
    try {
      process.kill(grandchildPid, 0); // signal 0 = existence check
      alive = true;
    } catch {
      // ESRCH = process doesn't exist = success
      alive = false;
    }
    expect(alive).toBe(false);
  }
});
```

Key design decisions:
- Uses `shellMode: "full"` to skip command allow-list checks (the command uses `wait` which is a shell builtin, not on the allow-list)
- Uses 500ms timeout — long enough for the echo to print the PID, short enough to trigger timeout
- The grandchild PID is captured from stdout via the `echo $!` in the bash command
- After timeout, waits 200ms for OS cleanup, then checks if grandchild is still alive using `process.kill(pid, 0)` (signal 0 is an existence check that throws ESRCH if process is gone)
- Skipped on Windows via `it.skipIf(process.platform === "win32")`
